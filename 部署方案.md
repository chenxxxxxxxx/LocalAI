# LocalAI + Intel SYCL llama.cpp 部署方案

## 1️⃣ LocalAI 后端架构（纠正）

LocalAI 后端通信协议是 **gRPC（Protocol Buffers）**，不是 JSON。

- 每个后端是一个独立的 **gRPC 服务器进程**，实现 `backend/backend.proto` 定义的 `Backend` service
- 核心 RPC 方法：`LoadModel`、`Predict`、`PredictStream`（流式）、`Embedding`、`Health` 等
- LocalAI 主进程通过 `pkg/grpc/client.go` 与后端通信，自动分配端口、启动进程、健康检查
- 后端可以用 C++、Go、Python 或任何支持 gRPC 的语言实现

> ⚠️ 任何用 shell 脚本包装 llama-cli 返回 JSON 的方案都**无法工作**，因为 LocalAI 严格依赖 gRPC 接口

### 已有的 Intel SYCL 支持

LocalAI **已原生支持** Intel SYCL，无需从零开发：

- `backend/index.yaml` 中定义了 `intel: "intel-sycl-f16-llama-cpp"` 能力映射
- `backend/Dockerfile.llama-cpp` 支持 `BUILD_TYPE=intel` / `sycl_f16` / `sycl_f32`
- `core/backend/options.go` 自动为 Intel SYCL 后端禁用 mmap（已知兼容性问题，见 issue #9012）
- `pkg/xsysinfo/gpu.go` 支持 Intel GPU 自动检测（通过 `xpu-smi` 或 `intel_gpu_top`）
- GPU 层数默认 `9999999`（全部卸载到 GPU），无需手动设置

### 多协议 API 支持（改造后自动获得）

LocalAI 同时兼容 **6 种 API 格式**，gRPC 后端集成后全部自动可用：

| 协议格式 | 关键端点 | 适用场景 |
|---------|---------|---------|
| **OpenAI** | `/v1/chat/completions`、`/v1/embeddings`、`/v1/audio/speech` | 最多生态支持，ChatGPT 客户端通用 |
| **Ollama** | `/api/chat`、`/api/generate`、`/api/embed` | Ollama 生态工具链兼容 |
| **Anthropic** | `/v1/messages` | Claude API 客户端兼容 |
| **Jina** | `/v1/rerank` | 文档重排序 / RAG 检索增强 |
| **ElevenLabs** | `/v1/text-to-speech/:voice-id` | TTS 语音合成 |
| **Open Responses + MCP** | `/v1/responses`、`/v1/mcp/chat/completions` | Agent 工作流、MCP 工具自动注入 |

> 这意味着：SYCL 后端接入后，你的 285H 不仅可以用 OpenAI 客户端调用，还能直接被 Ollama 工具、Claude 客户端等访问，无需额外适配。

## 2️⃣ 三种部署方案（按推荐优先级排列）

### 方案 A：Docker Compose 部署 Intel SYCL 镜像 ⭐推荐

最简单，不需要改任何代码。但有三个必要前提：

#### 前提条件

| 条件 | 说明 |
|------|------|
| **① 使用 `-gpu-intel` 专用镜像** | 项目默认 `docker-compose.yaml` 中的镜像是 CPU 版本，不含 SYCL 支持 |
| **② 挂载 `/dev/dri` 设备** | Intel 集显通过 DRI 设备暴露给容器，不挂载则无法访问 GPU |
| **③ 宿主机安装 Intel GPU 驱动** | Ubuntu 24.04 需要安装 i915 内核驱动和 GPU 运行时库 |

#### 可用镜像标签

| 描述 | Quay.io | Docker Hub |
|------|---------|------------|
| 稳定版 | `quay.io/go-skynet/local-ai:latest-gpu-intel` | `localai/localai:latest-gpu-intel` |
| 开发版 | `quay.io/go-skynet/local-ai:master-gpu-intel` | `localai/localai:master-gpu-intel` |
| 版本锁定 | `quay.io/go-skynet/local-ai:v4.1.3-gpu-intel` | `localai/localai:v4.1.3-gpu-intel` |

#### 宿主机准备（Ubuntu 24.04）

```bash
# 1. 安装 Intel GPU 驱动和运行时（Ubuntu 24.04 已内置 i915 内核驱动）
sudo apt-get update && sudo apt-get install -y \
  intel-media-va-driver mesa-vulkan-drivers intel-gpu-tools

# 2. 验证 GPU 设备可用
ls /dev/dri/          # 应看到 card0/card1、renderD128 等（Meteor Lake 可能显示 card1）
# ⚠️ 285H (Meteor Lake) 同时加载 i915(显示) + xe(计算) 两个驱动
# intel_gpu_top 需要较新版本才支持 xe，可用以下替代命令验证：
sudo intel_gpu_top    # 需要 root 或 render/video 组权限
# 或：xpu-smi discovery（需已安装 oneAPI 工具链）

# 3. 确认当前用户有 render 组权限
sudo usermod -aG render $USER && newgrp render
```

#### docker-compose.yaml 配置

```yaml
services:
  api:
    image: localai/localai:latest-gpu-intel
    ports:
      - 8080:8080
    env_file:
      - .env.intel-285h    # 285H 专用环境变量配置
    environment:
      - MODELS_PATH=/models
      - DEBUG=true
      # 285H 共享内存有限，建议开启单后端模式
      - LOCALAI_SINGLE_ACTIVE_BACKEND=true
      # 可选：强制 Intel 后端（跳过自动检测）
      # - LOCALAI_FORCE_META_BACKEND_CAPABILITY=intel
    devices:
      - /dev/dri           # ⚠️ 必须挂载，否则无法使用 Intel GPU
    volumes:
      - ./models:/models
      - ./generated/images:/tmp/generated/images
      - ./data:/data
      - backends:/backends       # 缓存后端镜像，避免重复下载
      - configuration:/configuration  # 持久化运行时配置
    # command:
    #   - gemma4-e4b       # 可选：指定启动时预加载的模型名称

volumes:
  backends:
  configuration:
```

#### .env.intel-285h 配置文件（Intel Core Ultra 9 285H 专用）

与 `docker-compose.yaml` 配合使用，通过 `env_file: .env.intel-285h` 自动加载：

```bash
## ===== Intel Core Ultra 9 285H 专用 .env =====

## CPU 线程数：285H 有 6P+8E+2LPE 核心，推荐设为 P-core 数量（性能核）
## E-core 效率较低，用于推理反而降速
LOCALAI_THREADS=6

## 默认上下文大小（集显共享内存有限，模型 YAML 可单独覆盖）
LOCALAI_CONTEXT_SIZE=4096

## 单后端模式：Arc 140T 共享系统内存，同时加载多模型易 OOM
LOCALAI_SINGLE_ACTIVE_BACKEND=true

## 后端切换时强制关闭旧后端（配合单后端模式释放显存）
LOCALAI_FORCE_BACKEND_SHUTDOWN=true

## API 监听地址（0.0.0.0 允许局域网访问）
LOCALAI_ADDRESS=0.0.0.0:8080

## 模型画廊（启用后可在 Web UI 浏览和一键安装模型）
LOCALAI_GALLERIES=[{"name":"localai", "url":"github:mudler/LocalAI/gallery/index.yaml@master"}]

## CORS（如需从浏览器或其他机器跨域调用 API）
LOCALAI_CORS=true
LOCALAI_CORS_ALLOW_ORIGINS=*

## 调试模式（初始部署建议开启，稳定后改为 info）
LOCALAI_LOG_LEVEL=debug

## 看门狗：防止后端推理卡死占用 GPU 资源
LOCALAI_WATCHDOG_BUSY=true
LOCALAI_WATCHDOG_BUSY_TIMEOUT=5m

## 空闲回收：15 分钟无请求自动卸载模型，释放共享显存
LOCALAI_WATCHDOG_IDLE=true
LOCALAI_WATCHDOG_IDLE_TIMEOUT=15m

## llama.cpp 并行请求数（共享显存有限，保持 1 避免 OOM）
LLAMACPP_PARALLEL=1

## 可选：强制使用 Intel SYCL 后端（跳过自动 GPU 检测）
## 通常不需要，系统自动检测 Arc 140T
# LOCALAI_FORCE_META_BACKEND_CAPABILITY=intel

## 可选：HuggingFace 模型缓存路径
# HUGGINGFACE_HUB_CACHE=/data/huggingface
```

> **285H 硬件说明**：Core Ultra 9 285H 的 Arc 140T 集显拥有 8 个 Xe 核心（2.35 GHz），与系统共享内存。以 32GB 系统内存为例，扣除系统占用后 GPU 可用约 12-16GB，足以运行 Q4_K_M 量化的 4B-8B 模型。

#### 启动与验证

```bash
# 启动
docker compose up -d

# 查看日志确认 Intel GPU 被检测到
docker compose logs -f api | grep -i "intel\|sycl\|gpu"

# 测试推理
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "your-model", "messages": [{"role": "user", "content": "Hello"}]}'
```

> ⚠️ **常见误区**：项目自带的 `docker-compose.yaml` 默认使用 `quay.io/go-skynet/local-ai:master`（CPU 镜像），即使运行在 Intel 硬件上也**不会启用 SYCL 加速**。必须换用 `-gpu-intel` 镜像。

适用场景：标准部署，不需要自定义编译参数

### 方案 B：本地编译 SYCL 版 gRPC 后端 + 外部后端注册

适用场景：需要自定义编译参数（如特定优化选项），或者已编译好 SYCL 版 llama.cpp

步骤：
1. 用 `BUILD_TYPE=sycl_f16` 编译 llama-cpp 的 **gRPC 服务器**（不是 llama-cli）
   ```bash
   source /opt/intel/oneapi/setvars.sh
   cd backend/cpp/llama-cpp
   BUILD_TYPE=sycl_f16 make llama-cpp-grpc
   ```
2. 通过 CLI 参数注册为外部后端：
   ```bash
   local-ai --external-grpc-backends "sycl-llama:/path/to/grpc-server"
   ```
3. 模型配置 YAML 中指定后端：
   ```yaml
   name: my-model
   backend: sycl-llama
   parameters:
     model: /path/to/model.gguf
   ```

LocalAI 会自动：分配空闲端口 → 启动 gRPC 进程 → 健康检查 → 加载模型

### 方案 C：远程 gRPC 服务（分离部署）

适用场景：后端和 LocalAI 主进程需要运行在不同机器上

1. 在 GPU 机器上独立运行 SYCL gRPC 后端，监听固定端口
2. 在 LocalAI 主进程注册远程地址：
   ```bash
   local-ai --external-grpc-backends "sycl-llama:192.168.x.x:50051"
   ```

## 3️⃣ 注意事项

### 线程与 GPU 内存
- 285H + Arc 集显共享内存有限
- GPU 层数默认全部卸载（`NGPULayers=9999999`），适合集显场景
- 线程数不要超过物理线程数，可在模型 YAML 中配置 `threads`
- 建议配置 `MaxActiveBackends=1`（单后端模式），避免共享内存不足

### mmap 自动处理
- 代码已自动处理：后端名含 `intel` 或 `sycl` 时自动禁用 mmap
- 无需手动配置

### oneAPI 环境
- Docker 方式（方案 A）：`-gpu-intel` 镜像已内置 oneAPI 运行时和 Intel GPU 库，**容器内无需额外配置**
- 本地编译方式（方案 B）：启动前需 `source /opt/intel/oneapi/setvars.sh`
- 宿主机只需 GPU **内核驱动**（i915），不需要安装完整的 oneAPI 开发套件

### 内存管理
- LocalAI 内置 LRU 淘汰机制和内存回收（`MemoryReclaimerEnabled`）
- 建议配置 `WatchDogBusyTimeout` 防止 GPU 推理卡死
- 可设置 `MemoryReclaimerThreshold=0.90` 在内存使用达 90% 时自动回收

### 调试
- 开启 `--debug` 查看详细日志
- 先用 `llama-cli` 验证 SYCL 加速正常（tok/s），再接入 LocalAI
- 启用 `--enable-backend-logging` 查看后端进程输出

## 4️⃣ Intel SYCL 模型配置 YAML 示例

> 将 YAML 文件放入 `./models/` 目录（对应容器内 `/models`），LocalAI 启动时自动加载。
> 后端无需显式指定 `intel-sycl-f16-llama-cpp`——系统通过 capability 自动检测 Intel GPU 并选择 SYCL 后端。

### 示例 1：通用对话模型（自动模板，推荐）

适用于 GGUF 格式的现代模型（Qwen3、Llama3、Phi-4 等），由 llama.cpp 自动解析模型内嵌的 chat_template：

```yaml
# models/qwen3-sycl.yaml
name: qwen3-sycl
backend: llama-cpp
parameters:
  model: huggingface://Qwen/Qwen3-8B-GGUF/qwen3-8b-q4_k_m.gguf

# Intel SYCL 关键配置
f16: true                    # FP16 key/value 缓存，减少显存占用
mmap: false                  # Intel GPU 必须禁用（系统会自动处理，显式写更清晰）
context_size: 4096           # 285H 共享显存有限，建议 4096；独显可提至 8192+
# gpu_layers: 99999999       # 默认值，全部层卸载到 GPU，无需手动设置

# 自动使用模型内嵌的 Jinja chat_template（现代 GGUF 模型均支持）
template:
  use_tokenizer_template: true

stopwords:
  - '<|im_end|>'
  - '<|endoftext|>'
```

### 示例 2：手动模板对话模型

适用于需要精确控制 prompt 格式的场景，或者旧版 GGUF 不含模板的模型：

```yaml
# models/llama3-sycl.yaml
name: llama3-sycl
backend: llama-cpp
parameters:
  model: /models/Meta-Llama-3-8B-Instruct-Q4_K_M.gguf
  temperature: 0.7
  top_p: 0.9

f16: true
mmap: false
context_size: 4096
threads: 4                   # 不超过 285H 物理核心数

template:
  chat_message: |
    <|start_header_id|>{{ .RoleName }}<|end_header_id|>

    {{ if .Content -}}
    {{.Content -}}
    {{ end -}}
    {{ if .FunctionCall -}}
    {{ toJson .FunctionCall -}}
    {{ end -}}
    <|eot_id|>
  chat: |
    {{.Input }}
    <|start_header_id|>assistant<|end_header_id|>
  completion: |
    {{.Input}}

stopwords:
  - '<|eot_id|>'
  - '<|end_of_text|>'
```

### 示例 3：Function Calling（工具调用）模型

适用于 Agent / MCP 工作流，支持结构化函数调用输出：

```yaml
# models/qwen3-fcall-sycl.yaml
name: qwen3-fcall-sycl
backend: llama-cpp
parameters:
  model: huggingface://Qwen/Qwen3-8B-GGUF/qwen3-8b-q4_k_m.gguf

f16: true
mmap: false
context_size: 8192           # 工具调用需要更大上下文（函数定义 + 多轮对话）

function:
  disable_no_action: true
  grammar:
    disable: true            # Qwen3 原生支持工具调用格式，禁用 grammar 约束
    parallel_calls: true
  json_regex_match:
    - "(?s)<tool_call>(.*?)</tool_call>"
    - "(?s)<tool_call>(.*)"

template:
  use_tokenizer_template: true
  chat_message: |
    <|im_start|>{{if eq .RoleName "tool" }}user{{else}}{{ .RoleName }}{{end}}
    {{ if eq .RoleName "tool" -}}
    <tool_response>
    {{ end -}}
    {{ if .Content -}}
    {{.Content }}
    {{ end -}}
    {{ if eq .RoleName "tool" -}}
    </tool_response>
    {{ end -}}
    {{ if .FunctionCall -}}
    <tool_call>
    {{toJson .FunctionCall}}
    </tool_call>
    {{ end -}}<|im_end|>
  function: |
    <|im_start|>system
    You are a function calling AI model. You may call one or more functions to assist with the user query. Here are the available tools:
    {{range .Functions}}
    {"type": "function", "function": {"name": "{{.Name}}", "description": "{{.Description}}", "parameters": {{toJson .Parameters}} }}
    {{end}}
    For each function call return a json object with function name and arguments: {"name": <function-name>, "arguments": <json-arguments-object>}
    <|im_end|>
    {{.Input -}}
    <|im_start|>assistant
  chat: |
    {{.Input -}}
    <|im_start|>assistant
  completion: |
    {{.Input}}

stopwords:
  - '<|im_end|>'
  - '<|endoftext|>'
```

### 示例 6：Gemma 4 E4B（边缘推理旗舰，推荐 285H 使用）

Gemma 4 E4B 是 Google 专为边缘设备优化的模型（4.5B 有效参数 / 8B 含嵌入层），支持 128K 上下文、thinking 推理、function calling 和多模态（文本+图像+音频）。Q4_K_M 量化约 9.6GB，非常适合 285H 集显。

#### 基础对话配置

```yaml
# models/gemma4-e4b-sycl.yaml
name: gemma4-e4b
backend: llama-cpp
parameters:
  model: huggingface://ggml-org/gemma-4-E4B-it-GGUF/gemma-4-E4B-it-Q4_K_M.gguf
  # Gemma 4 官方推荐采样参数
  temperature: 1.0
  top_p: 0.95
  top_k: 64

# Intel SYCL 关键配置
f16: true
mmap: false
context_size: 4096             # 285H 集显建议值；内存充裕可提至 8192

# 自动使用 Gemma 4 内置 chat_template
template:
  use_tokenizer_template: true

stopwords:
  - '<end_of_turn>'
  - '<start_of_turn>'
```

#### 启用 Thinking（深度推理）配置

Gemma 4 支持可控的 thinking 模式，在系统提示中加入 `<|think|>` 即可激活内部推理链：

```yaml
# models/gemma4-e4b-think.yaml
name: gemma4-e4b-think
backend: llama-cpp
parameters:
  model: huggingface://ggml-org/gemma-4-E4B-it-GGUF/gemma-4-E4B-it-Q4_K_M.gguf
  temperature: 1.0
  top_p: 0.95
  top_k: 64

f16: true
mmap: false
context_size: 8192             # thinking 模式生成更多 token，建议加大上下文

system_prompt: "<|think|>"      # 激活 Gemma 4 thinking 推理模式

template:
  use_tokenizer_template: true

# 推理链过滤（可选：提取思考过程但不返回给用户）
reasoning:
  regex:
    - '<\|channel>thought\n([\s\S]*?)<channel\|>'

stopwords:
  - '<end_of_turn>'
  - '<start_of_turn>'
```

> **Gemma 4 E4B vs Qwen3 8B 对比**：E4B 仅 4.5B 有效参数但 benchmark 接近 8B 模型水平（MMLU Pro 69.4%），且 GGUF 体积（9.6GB Q4_K_M）与 Qwen3 8B 相当。对 285H 来说两者都能流畅运行，E4B 在多模态和 thinking 方面更有优势。

### Intel SYCL 配置要点速查

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| `backend` | `llama-cpp` | 方案 A/C 使用；方案 B 填注册名 |
| `f16` | `true` | FP16 KV 缓存，减少约 50% 显存占用 |
| `mmap` | `false` | Intel GPU 不兼容内存映射（系统自动处理，显式写防止意外） |
| `context_size` | `4096`（285H 集显） | 共享显存有限；独显可 8192+ |
| `gpu_layers` | 不填（默认全部卸载） | 默认值 `99999999`，自动全部卸载到 GPU |
| `threads` | `4`~`8` | 不超过物理核心数 |
| `low_vram` | `true`（可选） | 显存紧张时启用，减少 KV 缓存占用 |
| `use_tokenizer_template` | `true` | 现代 GGUF 模型推荐，自动使用内嵌模板 |

## 5️⃣ 推荐执行流程

1. ✅ 编译 SYCL llama.cpp，确保 llama-cli 可运行（已完成）
2. 选择方案 A/B/C（推荐先试方案 A）
3. 如选方案 B：编译 gRPC 服务器（非 llama-cli），用 `--external-grpc-backends` 注册
4. 创建模型配置 YAML，指定 `backend` 和模型路径
5. 启动 LocalAI，验证多协议接口：
   - OpenAI：`POST /v1/chat/completions`
   - Ollama：`POST /api/chat`（如有 Ollama 客户端需求）
   - Anthropic：`POST /v1/messages`（如有 Claude 客户端需求）
6. 检查流式响应（`stream: true`）、工具调用等高级功能
