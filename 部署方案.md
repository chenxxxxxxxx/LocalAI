# LocalAI + Intel SYCL 部署方案（Intel Core Ultra 9 285H 专用）

> 面向 **Docker Compose 部署**的精简指南，基于 Intel Core Ultra 9 285H (Arc 140T, 96GB 内存) 实测整理。

---

## 1️⃣ 前提条件

| 条件 | 说明 |
|------|------|
| **使用 `-gpu-intel` 镜像** | 默认 CPU 镜像不含 SYCL 支持，必须用 `localai/localai:latest-gpu-intel` |
| **挂载 `/dev/dri` 设备** | Intel GPU 通过 DRI 设备暴露给容器 |
| **宿主机 i915/xe 内核驱动** | Ubuntu 24.04 已内置，确认驱动已加载 |
| **用户加入 render 组** | 保证容器可访问 GPU 设备文件 |

---

## 2️⃣ 宿主机准备（Ubuntu 24.04）

```bash
# 1. 安装 Intel GPU 运行时和诊断工具
sudo apt update && sudo apt install -y \
  intel-media-va-driver mesa-vulkan-drivers intel-gpu-tools \
  intel-opencl-icd clinfo

# 2. 验证 GPU 设备
ls /dev/dri/              # 应看到 card1, renderD128 等
clinfo | grep "Max memory allocation"  # 查看 Arc 140T 可用 GPU 内存

# 3. 用户加入 render 组
sudo usermod -aG render $USER
newgrp render
```

**285H 实测：** `clinfo` 输出 `Max memory allocation: ~46.93 GiB`，Arc 140T 通过 xe 驱动动态分配系统内存，无需 BIOS 设置。

## 4️⃣ docker-compose.yml

```yaml
services:
  local-ai:
    image: localai/localai:latest-gpu-intel
    ports:
      - 8080:8080
    env_file:
      - .env.intel-285h
    environment:
      - MODELS_PATH=/models
      - DEBUG=true
    devices:
      - /dev/dri           # Intel GPU 必须挂载
    volumes:
      - ./models:/models
      - ./data:/data
      - ./backends:/backends
      - ./configuration:/configuration
      - ./generated/images:/tmp/generated/images   # 图像生成输出（如用 stable-diffusion 后端）
    # 资源限制：保障 IDE/编译工具不受推理进程影响
    deploy:
      resources:
        limits:
          cpus: '10'
          memory: 40G
```

## 6️⃣ 启动与验证

### 步骤 1：启动容器

```bash
cd ~ && docker compose up -d
sleep 10
```

### 步骤 2：⚠️ 关键！手动安装 Intel SYCL 后端

> LocalAI `latest-gpu-intel` 镜像**不会自动安装** SYCL llama.cpp 后端。  
> 不装的话所有模型都回退到 CPU AVX2，虽然能跑但速度只有一半。

```bash
# 安装 Intel SYCL 版 llama.cpp 后端（~500MB，首次需下载）
curl -X POST http://localhost:8080/backends/apply \
  -H "Content-Type: application/json" \
  -d '{"id":"localai@intel-sycl-f16-llama-cpp"}'

# 等待安装完成（约 1-2 分钟）
docker compose logs -f local-ai | grep -iE "download|install|complete"
# 看到 "installation completed" 就 Ctrl+C

# 验证 SYCL 后端已就位
docker compose exec local-ai ls /backends/intel-sycl-f16-llama-cpp/lib/ | grep sycl
# 应看到：libmkl_sycl.so、libsycl.so 等
```

### 步骤 3：重启并验证 GPU 加速

```bash
docker compose restart local-ai
sleep 10

# 发一次请求触发模型加载
curl -s http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"qwen2.5-coder-7b","messages":[{"role":"user","content":"hi"}],"max_tokens":10}' > /dev/null

# ⚠️ 关键验证：查 offload 日志
docker compose logs local-ai --since 1m | grep -iE "SYCL0|offloaded.*GPU|using device"
```

**应看到以下行才算成功：**

```
using device SYCL0 (Intel(R) Arc(TM) Graphics) - XXXXX MiB free
offloaded 29/29 layers to GPU
load_tensors: SYCL0 model buffer size = XXXX MiB
```

如果只看到 `CPU model buffer size`，说明 SYCL 后端没生效，返回步骤 2 检查。

### 步骤 4：日常使用

```bash
# 列出模型
curl -s http://localhost:8080/v1/models | python3 -m json.tool

# 测试推理
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"qwen2.5-coder-7b","messages":[{"role":"user","content":"你好"}]}'

# 查速度
docker compose logs local-ai --since 2m | grep -E "tok/s|eval time"
```

## 7️⃣ 模型性能基准测试（批量跑速度）

新增 / 更换模型后，用以下脚本批量测真实 GPU 速度，写入对比表格。

### 基准测试脚本

```bash
# 用同一 prompt 测多个模型，保证可比
for model in qwen3.5-4b gemma-4-e4b-it qwen2.5-coder-7b qwen2.5-14b; do
  echo "=== Testing: $model ==="
  curl -s http://localhost:8080/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d "{\"model\":\"$model\",\"messages\":[{\"role\":\"user\",\"content\":\"写一个Go的快速排序并解释\"}]}" > /dev/null
  sleep 2
  docker compose logs local-ai --since 30s | grep -E "eval time.*tokens per second" | tail -2
  echo ""
done
```

### 结果解读

每个模型产出两行：

```
prompt eval time = 542.08 ms /  35 tokens ( 15.49 ms/tok,  64.57 tok/s)  ← 输入处理速度
      eval time = 45849 ms  / 573 tokens ( 80.02 ms/tok,  12.50 tok/s)  ← 生成速度（关键）
```

| 指标 | 含义 | 关注点 |
|------|------|-------|
| **prompt eval** | 处理输入 token 的速度（预填充）| 长 prompt 场景重要 |
| **eval time** | 生成输出 token 的速度 | **日常体感速度，最重要** |

### 判断 GPU 是否生效

**真 GPU 生成速度应满足（285H Arc 140T 基准）：**

| 参数量 | 最低合格 | 正常 | 异常信号 |
|--------|---------|------|---------|
| 4B | >12 tok/s | 15-17 | <10 说明走了 CPU |
| 7B | >10 tok/s | 12-14 | <8 说明走了 CPU |
| 14B | >5 tok/s | 6-7 | <4 说明配置错 |

生成速度**明显低于**上面的"正常"范围时，查日志：

```bash
docker compose logs local-ai --since 2m | grep -iE "SYCL0 model buffer|CPU model buffer"
```

- 看到 `SYCL0 model buffer` → GPU 正常
- 只看到 `CPU model buffer` → 回退 CPU，参考第 6 节步骤 2 重装 SYCL 后端

### 建议记录

每次新增模型或更换 GGUF 源后，**更新下面第 8 和 9 节的速度表**，方便后续对比和选型。

## 8️⃣ Intel SYCL 模型架构兼容性（实测 285H）

### ✅ 实测 GPU 加速速度（SYCL 后端安装后）

| 模型 | 参数 | Prompt tok/s | 生成 tok/s | 适用场景 |
|------|------|--------------|-------------|---------|
| `qwen3.5-4b` | 4B | 22.7 | **16.67** | 快速对话 / 推理（首选）|
| `gemma-4-e4b-it` | 4.5B active | 37.9 | **13.16** | 多模态 + 通用 |
| `qwen2.5-coder-7b` | 7B | 64.6 | **12.50** | 代码 |
| `qwen2.5-14b` | 14B | 28.2 | 6.65 | 长文本写作（速度受集显带宽限制）|

> **带宽观察**：Arc 140T 与 CPU 共享 DDR5 内存带宽（~90GB/s）。小模型（4-7B）GPU 收益明显；14B+ 级别权重太大，每 token 读取全部权重时带宽饱和，反而不如 CPU AVX2 快。

### ❌ 不兼容（回退 CPU，速度极慢）

| 架构 | 代表模型 | 症状 |
|------|---------|-----|
| Qwen2VL（视觉编码器）| Qwen2.5-VL-7B | 1.5 tok/s，encoder 走 CPU |
| DeltaNet / SSM 混合 | Qwen3.5-9B+ | 7 分钟/请求，几乎全 CPU |
| 超大模型（>30B）| Qwen3-Coder-Next | VRAM 不足以 offload，极慢 |

### 选型规律

- ✅ **标准 Transformer**（Qwen2.x / Llama / Gemma / Mistral）→ SYCL 完全支持
- ❌ **新混合架构**（DeltaNet、Mamba、线性注意力）→ SYCL 后端支持不全
- ❌ **VL 视觉模型** → 视觉 encoder 特殊架构导致 CPU 回退
- 📊 **推荐参数量**：4B-14B Q4_K_M 最佳

## 9️⃣ 推荐模型 YAML 配置

将 YAML 放入 `~/models/`，容器自动加载。关键参数统一设置：

- `f16: true` — FP16 KV 缓存，减少显存占用
- `mmap: false` — Intel GPU 必须禁用（系统自动处理，显式写更清晰）
- `gpu_layers: 9999999` — 全部层 offload 到 GPU
- `flash_attention: true` — 开启 Flash Attention，长上下文提速 10-20%
- `threads: 8` — 与 .env 统一
- `context_size` — 按用途调整（下详）

### 通用对话模型（Qwen2.5-14B 中文写作为例）

```yaml
name: qwen2.5-14b
backend: llama-cpp
parameters:
  model: Qwen2.5-14B-Instruct-abliterated-v2.Q4_K_M.gguf

f16: true
mmap: false
gpu_layers: 9999999
flash_attention: true
context_size: 16384    # 长篇写作建议 16K
threads: 8

template:
  use_tokenizer_template: true

stopwords:
  - "<|im_end|>"
```

### 代码模型（Qwen2.5-Coder-7B）

```yaml
name: qwen2.5-coder-7b
backend: llama-cpp
parameters:
  model: Qwen2.5-Coder-7B-Instruct-Q4_K_M.gguf

f16: true
mmap: false
gpu_layers: 9999999
flash_attention: true
context_size: 16384    # 7B 模型 KV cache 敏感，16K 平衡点
threads: 8

function:
  grammar:
    disable: true
  parallel_calls: true

template:
  use_tokenizer_template: true

stopwords:
  - "<|im_end|>"
```

### 多模态模型（Gemma 4 E4B，带视觉）

```yaml
name: gemma-4-e4b-it
backend: llama-cpp
parameters:
  model: gemma-4-E4B-it-Q4_K_M.gguf
  temperature: 0.8
  top_p: 0.95
  top_k: 64

mmproj: mmproj-gemma-4-E4B-it-bf16.gguf    # 视觉投影器

f16: true
mmap: false
gpu_layers: 9999999
flash_attention: true
context_size: 16384
threads: 8

function:
  automatic_tool_parsing_fallback: true
  grammar:
    disable: true

options:
  - use_jinja:true

template:
  use_tokenizer_template: true

stopwords:
  - "<end_of_turn>"
```

### 推理蒸馏模型（Qwen3.5-4B Claude 蒸馏）

```yaml
name: qwen3.5-4b
backend: llama-cpp
parameters:
  model: llama-cpp/models/Qwen3.5-4B.Q4_K_M.gguf    # 路径相对 /models
  temperature: 0.7
  top_p: 0.8
  top_k: 20

f16: true
mmap: false
gpu_layers: 9999999
flash_attention: true
context_size: 8192     # 推理链较长，8K 够用
threads: 8

function:
  grammar:
    disable: true

options:
  - use_jinja:true

template:
  use_tokenizer_template: true
```

## 1️⃣1️⃣ 常见问题

### Q1：模型加载后速度极慢（< 2 tok/s）
**原因**：模型架构在 Intel SYCL 上不被支持，层无法 offload。
**验证**：`docker compose logs local-ai | grep -i "offloaded\|sycl"`
**解决**：换成第 8 节中 ✅ 列表里的模型架构。

### Q2：模型不出现在 API 列表
- 确认 YAML 文件在 `~/models/` 下
- 确认 GGUF 文件名与 YAML 中 `parameters.model` 字段一致
- 重启容器：`docker compose restart local-ai`

### Q3：多个 YAML 配置了同一 `name`
LocalAI 会只加载后扫描到的那个，建议每个 YAML 用不同的 `name`。

### Q4：BIOS 不能设置集显 VRAM 怎么办？
**无需设置**。Arc 140T 使用 xe 驱动动态分配系统内存，`clinfo` 显示可用 ~47GB。

### Q5：如何查看实际 GPU 使用情况？
```bash
sudo intel_gpu_top          # 实时 GPU 负载
sudo cat /sys/kernel/debug/dri/1/clients   # 当前 GPU 客户端
cat /sys/class/drm/card1/device/tile0/gt0/freq0/act_freq_mhz  # GPU 当前频率
```

### Q6：如何切换模型？
LocalAI 在 `LOCALAI_SINGLE_ACTIVE_BACKEND=true` 模式下自动卸载旧模型、加载新模型。直接在 API 请求的 `model` 字段切换即可。

## 1️⃣2️⃣ 升级方案：加装外接独立 GPU（Arc A770）

### 背景

Arc 140T 集显共享 DDR5 内存带宽（~90 GB/s），跑 14B 以上模型被带宽瓶颈限制在 6-7 tok/s。外接 Arc A770 16GB 独显后：

- **带宽**：560 GB/s（提升 6 倍）
- **显存**：独立 16GB GDDR6（不占用系统内存）
- **14B 速度**：6.65 → ~25 tok/s
- **SYCL 生态零迁移**：LocalAI 配置几乎不改

### 硬件清单

| 物品 | 型号 | 价格 |
|------|------|------|
| 显卡坞 | 零刻 EX 拓展坞（内置 600W 电源）| ~1300 元 |
| 显卡 | 蓝戟 Arc A770 Photon 16GB | ~1999 元 |
| **合计** | | **~3300 元** |

> ⚠️ 零刻 EX 拓展坞**只兼容零刻特定型号**（GTi 系列 / SEi 系列旗舰款）。购买前确认机身有专用 PCIe 对接口。

### 步骤 1：硬件安装

1. **关机**并拔电源
2. 将 A770 插入零刻 EX 的 PCIe 插槽
3. 连接 A770 的 **2×8pin** 供电线（零刻 EX 附带）
4. 用附赠的**专用连接线**对接 EX 与主机背面的 PCIe 接口
5. 先插 EX 电源，后开主机电源

### 步骤 2：BIOS 设置（重要）

> 某些机型（如 185H）接外接显卡后 CPU 性能大幅下降，必须改这项。285H 建议也改。

1. 开机按 `Del` 进入 BIOS
2. `Advanced` → `System Agent (SA) Configuration` → `Graphics Configuration`
3. `Primary Display`: **`IGFX`**（默认 Auto 可能导致 CPU 降频）
4. `F4` 保存退出

### 步骤 3：Linux 驱动验证

```bash
# 1. 确认内核版本 ≥ 6.2（Ubuntu 24.04 默认 6.8 ✅）
uname -r

# 2. 确认 A770 被 PCI 总线识别
lspci | grep -i vga
# 应看到两个 Intel GPU：
# 00:02.0 VGA compatible controller: Intel Corporation (Arc 140T)
# 03:00.0 VGA compatible controller: Intel Corporation DG2 [Arc A770]

# 3. 确认 OpenCL / Level Zero 识别到双设备
sudo apt install -y intel-opencl-icd intel-level-zero-gpu level-zero clinfo
clinfo -l
# 应看到 2 个 Device：
# Platform #0: Intel(R) OpenCL Graphics
#   Device #0: Intel(R) Arc(TM) Graphics     ← 140T (level_zero:0)
#   Device #1: Intel(R) Arc(TM) A770 Graphics ← A770 (level_zero:1)

# 4. 查 A770 显存
clinfo | grep -A5 "A770" | grep "Global memory"
# 应显示约 15-16 GB
```

### 步骤 4：LocalAI 迁移（单卡方案，A770 跑所有模型）

修改 `.env.intel-285h`，指定使用 A770：

```bash
# 在 .env 末尾添加
# 使用外接 A770（level_zero:1）而非核显（level_zero:0）
ONEAPI_DEVICE_SELECTOR=level_zero:1
```

重启：

```bash
cd ~
docker compose restart local-ai

# 验证 LocalAI 使用 A770
docker compose logs local-ai | grep -i "using device"
# 应看到：using device SYCL0 (Intel(R) Arc(TM) A770 Graphics) - 15XXX MiB free
```

### 步骤 6（进阶）：双卡分工配置

如果想让 **A770 跑 LLM、140T 跑嵌入模型 / Whisper**（并行不抢资源），用**双 LocalAI 实例**：

#### docker-compose.yml 修改

```yaml
services:
  local-ai:              # 主 LLM（A770）
    image: localai/localai:latest-gpu-intel
    ports:
      - 8080:8080
    env_file:
      - .env.intel-285h
    environment:
      - MODELS_PATH=/models
      - DEBUG=true
      - ONEAPI_DEVICE_SELECTOR=level_zero:1   # 绑定 A770
    devices:
      - /dev/dri
    volumes:
      - ./models:/models
      - ./data:/data
      - ./backends:/backends
      - ./configuration:/configuration
    deploy:
      resources:
        limits:
          cpus: '10'
          memory: 40G

  local-ai-embed:        # 嵌入 / Whisper（140T）
    image: localai/localai:latest-gpu-intel
    ports:
      - 8081:8080        # 不同端口
    environment:
      - MODELS_PATH=/models
      - ONEAPI_DEVICE_SELECTOR=level_zero:0   # 绑定核显 140T
      - LOCALAI_SINGLE_ACTIVE_BACKEND=false   # 可并行加载
      - LLAMACPP_PARALLEL=2
    devices:
      - /dev/dri
    volumes:
      - ./models-embed:/models                 # 独立模型目录
      - ./data-embed:/data
      - ./backends-embed:/backends
    deploy:
      resources:
        limits:
          cpus: '4'
          memory: 16G
```

### 回滚方案

如果 A770 使用中出现问题，快速回退到核显：

```bash
# 1. 关闭拓展坞电源
# 2. 修改 .env.intel-285h，注释或改为 level_zero:0
sed -i 's/ONEAPI_DEVICE_SELECTOR=level_zero:1/# ONEAPI_DEVICE_SELECTOR=level_zero:0/' ~/.env.intel-285h

# 3. 重启
docker compose restart local-ai

# 即刻恢复到第 8 节的 140T 速度基准
```

### 故障排查

| 症状 | 可能原因 | 解决 |
|------|---------|------|
| `clinfo` 只看到 1 个 GPU | EX 电源未开 / 连接线松 | 检查硬件连接 |
| `lspci` 不显示 A770 | BIOS 未识别 / PCIe 链路异常 | 重启，进 BIOS 确认 |
| LocalAI 仍使用 140T | `ONEAPI_DEVICE_SELECTOR` 未生效 | 确认 env 文件已加载，`docker compose config` 查看 |
| A770 识别但速度仍慢 | SYCL 后端未重新加载 | `docker compose down && docker compose up -d` |
| CPU 性能下降 | BIOS `Primary Display` 未改 | 见步骤 2 |
