# LocalAI Intel SYCL 部署文档

> **目标平台**：Intel Core Ultra 9 285H + Arc 140T 集显 / Ubuntu 24.04
> **部署方式**：Docker Compose（方案 A）

---

## 1. 项目目录结构

部署前需要准备以下目录和文件：

```
LocalAI/
├── docker-compose.yaml          # 容器编排配置（已适配 Intel SYCL）
├── .env.intel-285h              # 环境变量配置（285H 专用）
├── models/                      # ⬅️ 模型配置 + 模型文件存放目录
│   ├── gemma4-e4b-sycl.yaml     #   模型配置文件（下文详述）
│   └── *.gguf                   #   模型权重文件（自动下载或手动放入）
├── data/                        # 运行时数据（自动创建）
└── generated/images/            # 生成的图片输出（自动创建）
```

### 需要手动创建的目录

```bash
mkdir -p models generated/images data
```

## 2. 宿主机环境准备

### 2.1 安装 Intel GPU 驱动

Ubuntu 24.04 已内置 i915 内核驱动，只需补装用户态运行时：

```bash
sudo apt-get update && sudo apt-get install -y \
  intel-media-va-driver mesa-vulkan-drivers intel-gpu-tools
```

### 2.2 验证 GPU 设备

```bash
# 确认 DRI 设备存在
ls /dev/dri/
# 预期输出：card0  renderD128（或类似）

# 确认 Intel GPU 可监控
intel_gpu_top
# 按 q 退出
```

### 2.3 用户权限

```bash
# 将当前用户加入 render 组（否则容器可能无权访问 GPU）
sudo usermod -aG render $USER
# 重新登录或执行：
newgrp render
```

### 2.4 Docker 环境

```bash
# 确认 Docker 和 Compose 插件已安装
docker --version          # 需要 20.10+
docker compose version    # 需要 v2.x
```

## 3. 配置文件详解

### 3.1 docker-compose.yaml

此文件已适配 285H 平台，直接使用即可。以下是每个关键配置的说明：

```yaml
services:
  api:
    image: localai/localai:latest-gpu-intel
    #       └── 必须使用 -gpu-intel 后缀的镜像，否则无 SYCL 加速
    ports:
      - 8080:8080            # API 服务端口，可改为其他端口如 8088:8080
    env_file:
      - .env.intel-285h      # ⬅️ 环境变量文件路径（见 3.2 节）
    environment:
      - MODELS_PATH=/models  # 容器内模型路径，与 volumes 映射对应
      - DEBUG=true
      - LOCALAI_SINGLE_ACTIVE_BACKEND=true   # 单后端模式（集显必须）
    devices:
      - /dev/dri             # ⬅️ 必须挂载！Intel GPU 通过此设备暴露给容器
    volumes:
      - ./models:/models     # 本地 models/ 目录 → 容器内 /models
      - ./generated/images:/tmp/generated/images
      - ./data:/data
      - backends:/backends          # Docker 命名卷，缓存后端二进制
      - configuration:/configuration  # Docker 命名卷，持久化运行时配置

volumes:
  backends:        # 后端二进制缓存（首次拉取后不再重复下载）
  configuration:   # 运行时配置持久化
```

**配置优先级**：`environment` 中的变量 > `.env.intel-285h` 中的变量。两处都设置时，`environment` 优先。

### 3.2 .env.intel-285h（环境变量配置）

此文件集中管理所有 LocalAI 运行时参数。**修改后需重启容器生效**。

| 变量名 | 当前值 | 作用 | 调整建议 |
|--------|--------|------|---------|
| `LOCALAI_THREADS` | `6` | CPU 推理线程数 | 285H 有 6 个 P-core，保持 6 即可；不要设为 16（E-core 会拖慢速度） |
| `LOCALAI_CONTEXT_SIZE` | `4096` | 全局默认上下文长度 | 集显建议 4096；模型 YAML 可单独覆盖此值 |
| `LOCALAI_SINGLE_ACTIVE_BACKEND` | `true` | 同时只运行一个后端 | **集显必须开启**，否则多模型同时加载会 OOM |
| `LOCALAI_FORCE_BACKEND_SHUTDOWN` | `true` | 切换模型时强制关闭旧后端 | 配合单后端模式，确保显存完全释放 |
| `LOCALAI_ADDRESS` | `0.0.0.0:8080` | API 监听地址 | `0.0.0.0` 允许局域网访问；仅本机用改为 `127.0.0.1:8080` |
| `LOCALAI_GALLERIES` | `[{...}]` | 模型画廊源 | 启用后可在 Web UI (`http://IP:8080`) 浏览和一键安装模型 |
| `LOCALAI_CORS` | `true` | 跨域访问 | 前端页面或其他机器调 API 时需要 |
| `LOCALAI_CORS_ALLOW_ORIGINS` | `*` | 允许的跨域来源 | 生产环境建议改为具体域名 |
| `LOCALAI_LOG_LEVEL` | `debug` | 日志级别 | 初始调试用 `debug`；稳定后改为 `info` 减少日志量 |
| `LOCALAI_WATCHDOG_BUSY` | `true` | 推理超时看门狗 | 防止模型推理卡死一直占用 GPU |
| `LOCALAI_WATCHDOG_BUSY_TIMEOUT` | `5m` | 推理超时阈值 | 超过 5 分钟无响应则杀掉后端 |
| `LOCALAI_WATCHDOG_IDLE` | `true` | 空闲回收看门狗 | 无请求时自动卸载模型，释放共享显存 |
| `LOCALAI_WATCHDOG_IDLE_TIMEOUT` | `15m` | 空闲回收阈值 | 15 分钟无请求则卸载；按需调整 |
| `LLAMACPP_PARALLEL` | `1` | llama.cpp 并行请求数 | 集显保持 1；独显可尝试 2-4 |

### 3.3 模型配置 YAML

**核心概念**：每个模型需要一个 YAML 配置文件，放在 `./models/` 目录下。LocalAI 启动时自动扫描加载。

#### YAML 文件命名

- 文件名即模型标识，如 `gemma4-e4b-sycl.yaml` 中的 `name: gemma4-e4b` 就是 API 调用时的模型名
- API 调用示例：`"model": "gemma4-e4b"`

#### YAML 字段说明

```yaml
# ========== 必填字段 ==========
name: gemma4-e4b              # 模型名称（API 调用时使用）
backend: llama-cpp            # 后端引擎（方案 A 固定填 llama-cpp）
parameters:
  model: huggingface://...    # 模型来源（见下方「模型来源格式」）

# ========== Intel SYCL 必填 ==========
f16: true                     # FP16 KV 缓存（减少约 50% 显存）
mmap: false                   # 禁用内存映射（Intel GPU 不兼容）

# ========== 推荐配置 ==========
context_size: 4096            # 上下文长度（覆盖 .env 中的全局值）
template:
  use_tokenizer_template: true  # 自动使用模型内嵌模板（现代 GGUF 推荐）

# ========== 可选配置 ==========
threads: 6                    # CPU 线程数（覆盖 .env 中的全局值）
# gpu_layers: 99999999        # GPU 层数（默认全部卸载，通常不需要填）
# low_vram: true              # 显存紧张时启用
stopwords:                    # 生成停止词（因模型而异）
  - '<end_of_turn>'
parameters:                   # 采样参数
  temperature: 1.0
  top_p: 0.95
  top_k: 64
```

#### 模型来源格式

| 格式 | 示例 | 说明 |
|------|------|------|
| **HuggingFace 自动下载** | `huggingface://ggml-org/gemma-4-E4B-it-GGUF/gemma-4-E4B-it-Q4_K_M.gguf` | 首次加载时自动下载到容器内缓存 |
| **本地文件路径** | `/models/my-model-Q4_K_M.gguf` | 手动下载后放入 `./models/` 目录 |
| **URL 下载** | `https://example.com/model.gguf` | 从任意 URL 下载 |

> **提示**：使用 HuggingFace 格式时，首次启动会下载模型文件（Gemma 4 E4B Q4_K_M 约 9.6GB），下载进度可在日志中查看。下载完成后会缓存，重启不会重复下载。

## 4. 模型配置示例

### 4.1 Gemma 4 E4B（推荐首选）

创建文件 `models/gemma4-e4b-sycl.yaml`：

```yaml
name: gemma4-e4b
backend: llama-cpp
parameters:
  model: huggingface://ggml-org/gemma-4-E4B-it-GGUF/gemma-4-E4B-it-Q4_K_M.gguf
  temperature: 1.0
  top_p: 0.95
  top_k: 64

f16: true
mmap: false
context_size: 4096

template:
  use_tokenizer_template: true

stopwords:
  - '<end_of_turn>'
  - '<start_of_turn>'
```

### 4.2 Gemma 4 E4B + Thinking 推理

创建文件 `models/gemma4-e4b-think.yaml`：

```yaml
name: gemma4-e4b-think
backend: llama-cpp
parameters:
  model: huggingface://ggml-org/gemma-4-E4B-it-GGUF/gemma-4-E4B-it-Q4_K_M.gguf
  temperature: 1.0
  top_p: 0.95
  top_k: 64

f16: true
mmap: false
context_size: 8192

system_prompt: "<|think|>"

template:
  use_tokenizer_template: true

reasoning:
  regex:
    - '<\|channel>thought\n([\s\S]*?)<channel\|>'

stopwords:
  - '<end_of_turn>'
  - '<start_of_turn>'
```

### 4.3 Qwen3 8B 通用对话

创建文件 `models/qwen3-sycl.yaml`：

```yaml
name: qwen3-sycl
backend: llama-cpp
parameters:
  model: huggingface://Qwen/Qwen3-8B-GGUF/qwen3-8b-q4_k_m.gguf

f16: true
mmap: false
context_size: 4096

template:
  use_tokenizer_template: true

stopwords:
  - '<|im_end|>'
  - '<|endoftext|>'
```

### 4.4 Embedding 向量模型（用于 RAG）

创建文件 `models/embedding-sycl.yaml`：

```yaml
name: text-embedding-sycl
backend: llama-cpp
embeddings: true
parameters:
  model: huggingface://CompendiumLabs/bge-base-en-v1.5-gguf/bge-base-en-v1.5-f16.gguf

f16: true
mmap: false
```

> **注意**：单后端模式下同一时间只能运行一个模型。如果加载了对话模型，再调用 Embedding 模型时，系统会自动卸载对话模型、加载 Embedding 模型（约需几秒切换时间）。

## 5. 部署步骤

### Step 1：准备目录和文件

```bash
cd /path/to/LocalAI

# 创建必要目录
mkdir -p models generated/images data

# 确认配置文件就位
ls docker-compose.yaml    # 容器编排
ls .env.intel-285h        # 环境变量
ls models/*.yaml          # 至少放一个模型 YAML
```

### Step 2：启动服务

```bash
docker compose up -d
```

### Step 3：查看启动日志

```bash
# 实时查看日志（Ctrl+C 退出）
docker compose logs -f api
```

**关注以下关键日志**：

```
# ✅ 成功标志 —— 看到类似以下内容表示 Intel GPU 已识别
[INFO] GPU vendor detected via binary vendor=Intel binary=intel_gpu_top
[INFO] Using intel capability

# ✅ 模型加载成功
[INFO] Model loaded successfully name=gemma4-e4b

# ❌ 失败标志 —— 如果看到以下内容需要排查
[WARN] VRAM is less than 4GB, defaulting to CPU    → GPU 未正确识别
[ERROR] failed to load model                        → 模型文件问题
```

可用过滤命令快速定位：

```bash
docker compose logs api | grep -i "intel\|sycl\|gpu\|error\|loaded"
```

### Step 4：验证 API

#### OpenAI 格式（最常用）

```bash
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma4-e4b",
    "messages": [{"role": "user", "content": "你好，请介绍一下你自己"}],
    "temperature": 1.0
  }'
```

#### 流式响应

```bash
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma4-e4b",
    "messages": [{"role": "user", "content": "写一首关于春天的诗"}],
    "stream": true
  }'
```

#### Ollama 格式

```bash
curl http://localhost:8080/api/chat \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemma4-e4b",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

#### Embedding 向量

```bash
curl http://localhost:8080/v1/embeddings \
  -H "Content-Type: application/json" \
  -d '{
    "model": "text-embedding-sycl",
    "input": "This is a test sentence"
  }'
```

#### 查看已加载模型

```bash
curl http://localhost:8080/v1/models | python3 -m json.tool
```

## 6. 日常运维

### 启停服务

```bash
# 启动（容器不存在则创建，已存在则启动）
docker compose up -d

# 停止（保留容器和数据）
docker compose stop

# 停止并删除容器（数据卷保留）
docker compose down

# 停止并删除容器 + 数据卷（⚠️ 清空后端缓存）
docker compose down -v
```

### 查看日志

```bash
# 实时日志
docker compose logs -f api

# 最近 100 行
docker compose logs --tail 100 api
```

### 更新镜像

```bash
docker compose pull     # 拉取最新镜像
docker compose up -d    # 用新镜像重建容器
```

### 修改配置后重启

```bash
# 修改 .env.intel-285h 或 docker-compose.yaml 后
docker compose down && docker compose up -d
```

### 添加新模型

```bash
# 1. 在 models/ 目录下创建新的 YAML 文件
# 2. 重启服务使其生效
docker compose restart api

# 或者不重启，通过 API 动态加载（如果模型 YAML 已存在）
curl http://localhost:8080/v1/chat/completions \
  -d '{"model": "新模型名称", "messages": [...]}'
# LocalAI 会自动发现并加载 models/ 目录下匹配的 YAML
```

## 7. 常见问题

### Q1：启动后日志显示 "defaulting to CPU"

**原因**：容器未识别到 Intel GPU。

**排查步骤**：
```bash
# 1. 确认宿主机 GPU 设备存在
ls -la /dev/dri/

# 2. 确认 docker-compose.yaml 中有 devices 挂载
grep -A1 "devices" docker-compose.yaml

# 3. 确认镜像是 -gpu-intel 版本（不是默认 CPU 镜像）
docker compose config | grep image
```

### Q2：模型下载很慢或失败

**解决方案**：手动下载 GGUF 文件后放入 `./models/` 目录，然后将 YAML 中的 `model` 改为本地路径：

```yaml
parameters:
  # 改前（自动下载）：
  # model: huggingface://ggml-org/gemma-4-E4B-it-GGUF/gemma-4-E4B-it-Q4_K_M.gguf
  # 改后（本地文件）：
  model: /models/gemma-4-E4B-it-Q4_K_M.gguf
```

### Q3：推理时报 OOM（内存不足）

**解决方案**（按优先级尝试）：
1. 确认 `.env.intel-285h` 中 `LOCALAI_SINGLE_ACTIVE_BACKEND=true`
2. 减小模型 YAML 中的 `context_size`（如 4096 → 2048）
3. 在模型 YAML 中添加 `low_vram: true`
4. 换用更小的量化版本（如 Q4_K_M → Q3_K_S）

### Q4：如何同时使用多个模型？

285H 集显下**不建议**同时加载多个模型。但可以通过 API 切换：

```bash
# 先调用模型 A（自动加载）
curl ... -d '{"model": "gemma4-e4b", ...}'

# 再调用模型 B（自动卸载 A，加载 B，约需几秒）
curl ... -d '{"model": "qwen3-sycl", ...}'
```

切换过程对客户端透明，只是首次切换时有几秒延迟。

### Q5：如何连接 Web UI？

LocalAI 内置 React Web UI，启动后直接访问：

```
http://服务器IP:8080
```

可在 Web UI 中：
- 对话测试
- 浏览和安装模型画廊中的模型
- 查看已加载模型状态
